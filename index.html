Quick Concept

Every time you declare a variable like int x = 5;, the compiler writes it into a Symbol Table — recording its name, type, scope, and memory address. This table is the compiler's memory.

Without it, a compiler cannot verify your code is correct or generate reliable output.

What is a Symbol Table?

A data structure the compiler builds while reading your source code — like an organised notebook.

A Symbol Table is a data structure the compiler builds throughout the process of reading source code. Every time it encounters a new declaration — a variable, function, or parameter — it records it with four key attributes.

The Compiler Without a Symbol Table

Without a symbol table, a compiler would be working completely blind — unable to verify if a variable was declared, check its type, or find its memory location. Every check it performs relies on this table.

What Happens at Each Stage — In Detail

Stage 1 · Lexical Analysis — Scanning & Tokenising

The compiler reads the program character by character and groups them into tokens. A token is the smallest meaningful unit — a keyword like int, a name like x, an operator like =, or a literal like 5. The lexer also strips out whitespace and comments, which carry no semantic meaning.

int x = 5; // tokens: [KEYWORD:int] [IDENT:x] [OP:=] [NUM:5] [SEMI:;]

Stage 2 · Parsing — Understanding Structure

The parser receives the token stream and applies grammar rules to determine what the tokens mean together. It builds a parse tree (or Abstract Syntax Tree) that represents the program's structure. Crucially, it distinguishes between a declaration (int x = 5;) and a usage (x + 1). Only declarations trigger a write to the symbol table.

The parser also tracks nesting — entering a function body pushes a new scope, and exiting it pops that scope back off. This scope stack is what makes local variables possible.

Stage 3 · Storing Identifiers — Populating the Table

Each time a declaration is confirmed, the compiler immediately inserts a new row into the symbol table with: the identifier's name, its data type, the scope level it was declared in, and the memory address allocated for it. This happens in one atomic operation so the table is always consistent.

If the identifier already exists in the current scope, the compiler raises a redeclaration error rather than overwriting the existing entry.

Stage 4 · Displaying the Table — Inspection & Debugging

After parsing completes, the full symbol table can be rendered in a structured tabular form — one row per identifier, columns for every attribute. This output is invaluable for debugging: developers and tools can inspect exactly which identifiers were found, at which scope, and where they live in memory. Many IDEs expose this as a "symbol browser."

Stage 5 · Semantic Analysis — Using the Table to Catch Errors

This is where the symbol table pays off. Every time an identifier is used (not declared), the compiler looks it up. If it is absent → undeclared error. If the type is incompatible with the operation → type mismatch. If it exists but in a different scope → scope violation. These checks are O(1) lookups because the symbol table is typically implemented as a hash table for constant-time access.

Semantic analysis runs hand-in-hand with parsing — errors are surfaced before any machine code is ever emitted, saving runtime crashes entirely.

Scope Management — A Closer Look

Scope is arguably the most complex attribute in the symbol table. It determines not just where an identifier was declared, but when it comes into existence and when it ceases to exist. Compilers manage scope using a structure called a scope stack.

The Scope Stack

Every time the compiler enters a new block — a function body, an if statement, a loop — it pushes a new scope onto the stack. Every identifier declared inside that block is registered only in that scope. When the block ends, the scope is popped, and all identifiers declared within it are no longer accessible. The symbol table entries remain (for error reporting purposes) but are marked inactive.

int x = 10; // scope: Global ← pushed to stack level 0

void add(int a, int b) { // scope: Function:add ← stack level 1
  int result = a + b; // scope: Function:add ← stack level 1
  return result;
} // ← stack level 1 popped; result, a, b gone

int r = result; // ❌ result is out of scope here

The Three Scope Levels

Most compiled languages use at least three layers of scope, all tracked in the symbol table:

Scope Level Where Declared Who Can Access Memory Region
Global Outside all functions Any function, any file Data segment (heap-adjacent)
Function / Local Inside a function body Only within that function Stack frame (temporary)
Parameter Function signature Only within that function Stack frame (passed by caller)

Shadowing — When Names Collide Across Scopes

A variable in an inner scope can share the same name as one in an outer scope. This is called shadowing. The symbol table resolves this by always searching the innermost active scope first. The outer variable is temporarily hidden, but not destroyed — it becomes visible again once the inner scope exits.

int x = 100; // Global x — address 0x1000

void foo() {
  int x = 42; // Local x — shadows global x
  print(x); // prints 42, not 100
}

print(x); // back to global — prints 100

The symbol table keeps both entries for x — distinguished by their scope column — so the compiler always knows which one is active at any given point.

Block Scope — Nested Scopes Inside Functions

Scope isn't limited to functions. Every pair of curly braces { } creates a new block scope — including loops and conditionals. The symbol table tracks the nesting depth, sometimes called the scope level or depth counter.

void example() { // scope depth 1
  int total = 0;

  for (int i = 0; i < 10; i++) { // scope depth 2 — i lives here
    int temp = i * 2; // scope depth 2 — temp lives here
    total += temp;
  } // depth 2 popped — i and temp gone

  print(i); // ❌ i is out of scope
}
Section 4

Why It Matters — Semantic Analysis

The symbol table turns the compiler from a grammar checker into a fully intelligent code verifier. Click each error type to expand the example and explanation.

🔴 Undeclared Identifier

The compiler looks up every identifier it encounters in the symbol table. If it finds no entry, it immediately raises an undeclared identifier error — the program cannot continue compilation.

int x = 5;
int y = 10;
int result = x + z; // ❌ z was never declared

What the compiler does:

1. Encounters z on line 3 during semantic analysis.
2. Looks up z in the symbol table — no entry found at any scope level.
3. Raises: error: 'z' undeclared
4. Compilation halts — no output is generated.

This prevents runtime crashes from reading uninitialised memory — a security risk in languages like C.

🟡 Redeclaration Error

When the compiler attempts to insert a new identifier into the symbol table, it first checks whether that name already exists in the current scope. If it does, declaring it again is an error — two entries with the same name would create ambiguity.

int x = 5;
int x = 10; // ❌ x already in symbol table at this scope

// But this is fine — different scope:
void foo() {
  int x = 99; // ✅ new scope, new entry allowed
}

What the compiler does:

1. Sees int x = 10; and attempts to insert x into the global scope.
2. Finds x already exists at scope level 0.
3. Raises: error: redeclaration of 'x'
4. Note: the same name in a different scope (like inside foo()) is allowed — it gets its own separate row in the table.

🟣 Type Mismatch

The Type column in the symbol table lets the compiler verify that every operation involves compatible data types. Adding an integer to a string, or assigning a float to a boolean, are all caught here — before a single byte of machine code is emitted.

int x = 5;
string s = "hello";
int r = x + s; // ❌ int + string — incompatible types

// Another common mismatch:
float f = 3.14;
bool b = f; // ❌ cannot assign float to bool

What the compiler does:

1. Evaluates the expression x + s.
2. Looks up x → type: int. Looks up s → type: string.
3. Checks arithmetic rules — int + string is not permitted.
4. Raises: error: invalid operands to binary +

Some languages (like JavaScript) allow implicit type coercion instead of erroring — but compiled languages like C and Java enforce strict type safety here.

🟢 Scope Violation

The Scope column records exactly where each identifier is valid. If code tries to access a variable outside its declared scope, the compiler catches it — even if a variable with that name exists elsewhere.

void add(int a, int b) {
  int result = a + b;
}

int r = result; // ❌ result is scoped to add() only
int s = a + 1; // ❌ a is a parameter of add(), not accessible here

What the compiler does:

1. Encounters result outside of add().
2. Searches active scope stack — only global scope is active at this point.
3. Finds result in the table but its scope is Function: add, which is currently inactive.
4. Raises: error: 'result' undeclared in this scope

Scope violations protect encapsulation — a function's internal variables are its own business and cannot leak outward.

Real-World Compiler Output Examples

This is what these errors actually look like when a real compiler (GCC / Clang) runs — the symbol table data maps directly to each message.

🔴 GCC — Undeclared Identifier Output

$ gcc program.c
program.c:3:22: error: 'z' undeclared (first use in this function)
3 | int result = x + z;
| ^
program.c:3:22: note: each undeclared identifier is reported only once

The caret ^ points exactly to the unknown identifier. The compiler knows this because the symbol table has no entry for z, so it can report line and column with precision.

🟡 GCC — Redeclaration Output

$ gcc program.c
program.c:2:7: error: redeclaration of 'x' with no linkage
2 | int x = 10;
| ^
program.c:1:7: note: previous declaration of 'x' was here
1 | int x = 5;
| ^

The compiler shows both the conflicting declaration and where the original was first recorded — directly from the symbol table's line number column.

🟣 GCC — Type Mismatch Output

$ gcc program.c
program.c:4:18: error: invalid operands to binary + (have 'int' and 'char *')
4 | int r = x + s;
| ^~~

The error message includes both type names — int and char * — which were retrieved directly from the Type column of the symbol table entries for x and s.

🟢 GCC — Scope Violation Output

$ gcc program.c
program.c:7:11: error: 'result' undeclared (first use in this function)
7 | int r = result;
| ^~~~~~
program.c:3:7: note: 'result' was declared here
3 | int result = a + b;
| ^~~~~~

Even though result exists in the symbol table, its Scope column shows it belongs to Function: add. The compiler's note helpfully points to where it was declared — pulled from the Line Number column.

The Bottom Line

Every compiler error message you have ever read — with its line number, variable name, and type information — is powered by data sitting in a symbol table. The table is not a passive record. It is an active, constantly-queried structure that makes compilers intelligent.
